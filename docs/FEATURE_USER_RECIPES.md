# Мини-план: «Своя книга рецептов» (первая фича Фазы B)

> **Версия:** 0.5 · **Создано:** 2026-05-22 · **Обновлено:** 2026-06-21 (§9 — находка про приватность из UX-теста) · **Статус: Шаги 1–8 написаны (вся фича в коде). Локально зелёные tsc + lint + паритет i18n; финальный прод-билд прогнать на Mac.**
>
> Решения Дарьи: **полный формат** (переиспользуем механику сайта) + **приватно** (v1).
> Это первая фича из Фазы B в `MONETIZATION_PLAN.md` (строим ценность; оплаты ещё нет).

---

## 1. Что строим (scope v1)

Пользователь в кабинете может: видеть список своих рецептов, **создать / редактировать /
посмотреть (красивая страница) / удалить**. Рецепты **приватные** — видит только их автор.
Поля — **полные**, как у авторских рецептов. Лимит Free — **15 рецептов** (под фиче-флагом,
пока выключен).

**Не входит в эту фичу:** КБЖУ (это следующий шаг — AI-КБЖУ посчитает и для пользовательских),
шеринг по ссылке, любой AI (генерация/картинки), оплата.

---

## 2. Модель данных (минимальная миграция, additive)

**`recipes`** — добавить:
- `owner_id uuid references auth.users(id) on delete cascade` — `null` у авторских/админских,
  = пользователь у пользовательских.
- `visibility text not null default 'public'` — `'public'` (каталог автора) | `'private'`
  (книга пользователя). Задел под `'unlisted'` (ссылка) на будущее.
- индекс на `owner_id`.

**`profiles`** — добавить тонкий монетизационный каркас:
- `plan text not null default 'free'` — `'free' | 'premium' | 'lifetime'`.

**`steps`, `recipe_categories`, storage-бакеты** — без изменений (переиспользуем).

**Slug:** колонка глобально `unique`. Чтобы пользовательские названия не конфликтовали с
авторскими — для пользовательских генерим `toSlug(title)-<rand>`, а в кабинете адресуем рецепт
по `id`, не по slug. (Альтернатива на будущее: частичный unique-индекс «`slug` уникален только
при `visibility='public'`».)

---

## 3. RLS (безопасность — самое важное)

Сейчас: `recipes` select = `published OR admin`; write = admin. Добавляем (пользователь управляет
**только своими**, шаблон — как в `migration-user-notes.sql`):

- **select:** `owner_id = auth.uid()` (видеть свои приватные) — в дополнение к существующему.
- **insert:** `with check (owner_id = auth.uid() AND visibility = 'private' AND published = false)`
  — пользователь физически не может выложить рецепт в публичный каталог.
- **update / delete:** `using (owner_id = auth.uid())`.
- **`steps` и `recipe_categories`:** политики через `exists (… recipes где owner_id = auth.uid())`,
  чтобы пользователь управлял шагами/категориями только своих рецептов.

---

## 4. Лимит 15 (entitlements-каркас)

- Серверный helper `getEntitlements(userId)` → `{ plan, limits: { recipes: 15 | ∞ } }`.
- Создание рецепта — через **server action**, которая ПЕРЕД вставкой считает `count(owner_id) < limit`,
  иначе мягкая ошибка «достигнут лимит Free, безлимит в Premium».
- За фиче-флагом **`MONETIZATION_ENABLED`** (env): пока `false` → лимит не применяется (пре-лонч,
  тестируешь свободно). Код готов, рубильник включаем на этапе платежей.
- **Запись пользовательских рецептов — только через server actions** (не клиентский insert):
  сервер форсит `owner_id` / `visibility='private'` / `published=false` и считает лимит. Клиент
  это подделать не может.

---

## 5. UI (в `/dashboard`)

- `/dashboard/recipes` — «Моя книга»: список карточек, кнопка «Добавить рецепт», счётчик `N / 15`
  (когда флаг включён).
- `/dashboard/recipes/new` — форма создания.
- `/dashboard/recipes/[id]/edit` — форма редактирования.
- `/dashboard/recipes/[id]` — просмотр (переиспользуем презентационные блоки страницы рецепта:
  шапка, состав, шаги; КБЖУ-блок появится со следующей фичей).
- Пункт в `DashboardNav`: «Моя книга».
- **Форма — лёгкая версия `RecipeForm`:** переиспользуем секции (поля, шаги, категории, загрузка
  фото, время/порции, заметка), но **без админских контролов** (`published`/`featured`, AI-перевод,
  AI-обложка, кнопки КБЖУ). Вариант: либо `RecipeForm` с пропом `mode: 'user' | 'admin'`, либо
  отдельная `UserRecipeForm` на общих под-компонентах.

---

## 6. Переиспользование (минимум нового кода)

- `toSlug`, `uploadFile`, бакеты `recipe-covers` / `step-photos` — как есть.
- `createRecipe` / `updateRecipe` (admin, клиентские, ставят published/featured) — **не переиспользуем
  напрямую**; делаем серверные user-экшены, но повторяем структуру (загрузка обложки, upsert шагов,
  категории).
- `RecipeCard`, презентация шагов/состава — переиспользуем для списка и просмотра.

---

## 7. Storage / приватность фото

Бакеты публичные → URL фото пользователя технически доступен по прямой (неугадываемой) ссылке.
Для v1 приемлемо. Будущее усиление: приватный бакет + signed URLs → в бэклог.

   (4 блоками из-за обрезки при вставке). Проверено: публичный сайт и шаги рецептов работают.
2. ✅ **Типы:** `Recipe` +`owner_id`/`visibility`; `Profile` +`plan`. tsc прошёл.
3. ✅ **`getEntitlements()`** + фиче-флаг `MONETIZATION_ENABLED`. Готово (`src/lib/entitlements.ts`):
   `getEntitlements(userId)` читает `profiles.plan` через service-role → `{ plan, monetizationEnabled,
   limits: { recipes: 15|null } }`; флаг выключен → лимит `null` (безлимит). Доп. хелпер
   `canCreateRecipe(userId, currentCount)` для Шага 4. Флаг задокументирован в `.env.example`. tsc прошёл.
4. ✅ **Server actions** (`src/app/dashboard/recipes/actions.ts`): `createUserRecipe` /
   `updateUserRecipe` / `deleteUserRecipe`. Сервер форсит `owner_id=auth.uid()` / `visibility='private'`

## 9. Открытые мелочи (по ходу, не блокеры)

- Лимит фото 3/рецепт (Free, стратегия §3) — можно отложить, минорный гейт.
- Просмотр рецепта, возможно, потребует лёгкого рефактора — вынести презентацию из `[slug]/page`
  в общий компонент, чтобы переиспользовать и в кабинете.
- **Приватность реализована верно, но не считывается пользователем (UX-тест, 2026-06-19).**
  RLS/`visibility='private'` работают как задумано (§3) — никто посторонний не видит чужую «свою
  книгу». Но один тестер из закрытой беты, близкий к ЦА, по умолчанию воспринял «свою книгу» как
  полу-публичное пространство для круга друзей/соцсеть, и понял, что это полностью приватно,
  только после прямого объяснения интервьюера (другой тестер до этого тоже сомневался — «готовая
  книга или своя?»). Реализация не требует изменений, но нужна явная одна строка копирайта в
  форме/онбординге создания рецепта (например «эти рецепты видишь только ты») — сейчас это нигде
  не сказано прямым текстом. Подробности — `../ux-testing/UX_Testing_Tracker.xlsx`, находка F79.
