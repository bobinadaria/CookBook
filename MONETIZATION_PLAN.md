# Монетизация — дизайн-документ (Этап 4)

> **Версия:** 0.2 · **Создано:** 2026-05-22 · **Обновлено:** 2026-05-24 · **Статус: ДИЗАЙН, не реализация.**
>
> Это чертёж под `CLAUDE.md` §2.4 и `PRODUCT_STRATEGY.md` §§3–7. Задача документа: зафиксировать
> модель, схему БД и порядок работ, чтобы потом строить без переделок. Числа — гипотезы (±30%),
> не факты.
>
> **Когда монетизация (обновлено 2026-05-24).** Раньше тут стояло «не раньше ~месяца 10». Это
> число устарело: «месяц 10» был оценкой *срока стройки* премиум-ценности при обычном темпе, а
> вайб-кодинг стройку перемотал. Реальный гейт — не календарь, а **два глагола**: *построить*
> монетизацию (Paddle, кредиты, paywall в коде) можно хоть скоро, всё под флагом, риск нулевой;
> *включить* её (`MONETIZATION_ENABLED=true`) — только когда есть **аудитория** (промо проведено,
> retention доказан — гейт Фазы 1 в `PRODUCT_STRATEGY` §7). Деньги идут по часам рынка, не по
> часам стройки. См. §8.

## 0. Главный вывод (читать первым)

**Порядок незыблем: сначала ценность → потом счётчики/гейты → в самом конце платёж.** Но
картина «гейтить пока нечего» (так было на 2026-05-22) уже сдвинулась — ценность строится
быстро. Статус на 2026-05-24:

- ✅ каталог рецептов автора, поиск/фильтры; избранное, профиль; тёмная тема, двуязычие;
- ✅ **своя книга рецептов** (приватные пользовательские рецепты, лимит под флагом) — это был
  «главный gate»; теперь построен;
- ✅ **AI-КБЖУ для пользователя** (главный AI-крючок) — через `getEntitlements`, под план-гейтом;
- 🔧 ещё не построено: генерация рецепта по запросу, импорт по URL, экспорт в PDF, меню недели,
  AI-картинки;
- 🔧 биллингового фундамента в БД пока нет: таблиц `subscriptions` / `credit_ledger` не
  существует (есть только `getEntitlements` + выключенный флаг `MONETIZATION_ENABLED`).

**Вывод (уточнён).** Стройка ценности обгоняет план — это часы стройки, и их вайб-кодинг
ускорил. Но включение оплаты упирается не в код, а в **аудиторию** (часы рынка): paywall
конвертирует только когда в воронке есть кому платить. Поэтому «месяц 10» как дату убираем, а
гейт переформулируем через условия (см. §8). Порядок работ — там же.

## 2. Кредиты и счётчики

- **Welcome Credits:** 10 КБЖУ + 5 рецептов, с time-based раскрытием — день 0: 3 КБЖУ + 1 рецепт;
  день 3: +2 КБЖУ + 1 рецепт; день 7: +5 КБЖУ + 3 рецепта. Навсегда (даёт повод вернуться в дни 3 и 7).
- **Ежемесячно:** +1 КБЖУ + 1 рецепт (retention + регулярные «встречи» с paywall).
- **AI-картинки:** только за кредиты (паки S/M/L). **Никогда по подписке** — защита от перерасхода.
- **Soft-cap:** 200 AI-действий/сутки даже для Premium (антиабьюз).
- **Hard-cap** на Free-AI (отдельный ключ, бюджет ~$50/мес) + budget-alert (~$300/мес).

**Принцип:** кредиты списываются **только на сервере, атомарно**. Клиент никогда не уменьшает баланс.

## 3. Схема БД (что добавить)

Сейчас (RLS): `profiles`, `categories`, `recipes`, `steps`, `recipe_categories`, `favorites`,
`user_notes`, `ingredients_base`. Ничего платёжного нет. Добавляем (миграции `scripts/*.sql`, **RLS обязательно**):

### 3.1 `profiles` — расширить
- `plan text not null default 'free'` — `'free' | 'premium' | 'lifetime'` (денорм-кеш для быстрых чтений)
- `plan_expires_at timestamptz null` — `null` = free/lifetime; для monthly/yearly — конец оплаченного периода
- `paddle_customer_id text null`

### 3.2 `subscriptions` — источник истины по биллингу (синк из Paddle)
`user_id` (pk, fk profiles) · `paddle_subscription_id` · `plan` (premium_monthly|premium_yearly|lifetime) ·
`status` (active|trialing|past_due|canceled) · `current_period_end timestamptz null` ·
`cancel_at_period_end bool` · `created_at` · `updated_at`.
**RLS:** SELECT только своё; запись — только service-role (вебхук).

### 3.3 `credit_ledger` — append-only, источник истины по кредитам
`id` · `user_id` · `kind` (`kbju|recipe|image`) · `delta int` (грант `+N`, расход `−1`) ·
`reason` (`welcome_day0|welcome_day3|welcome_day7|monthly|pack_S|consume_kbju|…`) · `created_at`.
Баланс = `SUM(delta)` по `kind`. При желании — кеш в `credit_balances`.
**RLS:** SELECT только своё; INSERT — только server-side. Списание — через `SECURITY DEFINER`
функцию `spend_credit(user, kind)`: атомарно проверяет баланс ≥ 1 и пишет `−1`, иначе ошибка
(защита от двойной траты и от клиента).

### 3.4 Soft-cap
Считаем из `credit_ledger`: `COUNT` расходов за сутки (отдельная таблица не нужна).

## 4. Логика тарифов (enforcement)

- Один серверный helper `getEntitlements(userId)` → `{ plan, isPremium, limits:{recipes,favorites}, credits:{kbju,recipe,image} }`. Кеш на запрос.
- **Все гейты — на сервере** (route handler / server action), перед действием. Клиенту не доверять.
- Глобальный фиче-флаг **`MONETIZATION_ENABLED`** (env). Пока `false` — лимиты/гейты/paywall
  не показываем (как Phase 1 стратегии: упор в лимит ведёт к «приходи завтра», не к покупке).
- **Paywall-UX:** упор в лимит — мягкий экран «безлимит за €7.90/мес», не глухая стена.

## 5. Paddle — чек-лист интеграции (когда дойдём — гейт см. §8)

1. Аккаунт Paddle (сначала **sandbox**), привязать чешское ИП (MOR — берёт VAT по ЕС на себя).
2. Создать продукты/цены: Premium monthly €7.90, Premium yearly €69, Lifetime €79 (one-time),
   паки S €4.90 / M €9.90 / L €19.90. Сохранить `price_id` → конфиг-маппинг `price_id → {plan|credits}`.
3. **Checkout:** Paddle.js overlay по CTA на `/pricing`; передавать `user id` + email в custom data.
4. **Вебхук** `/api/webhooks/paddle`: проверять подпись (`PADDLE_WEBHOOK_SECRET`); обрабатывать
   `subscription.created/updated/canceled` (→ `subscriptions` + `profiles.plan/expires`) и
   `transaction.completed` (→ lifetime / зачисление кредитов в `credit_ledger`). Идемпотентность по event id.
5. **Секреты** (env, не в клиент, кроме client-token): `PADDLE_API_KEY`, `PADDLE_WEBHOOK_SECRET`, `PADDLE_CLIENT_TOKEN`.
6. Тест в sandbox (тест-карты Paddle) → go-live флаг.
7. Налоги: финализировать с чешским tax-advisor (€100–200 разово). Paddle закрывает VAT ЕС; доход от Paddle декларируем.

### 5.1 Paddle vs Lemon Squeezy — анализ и рекомендация (2026-05-22)

**Рекомендация: Paddle** — для твоего кейса (чешское ИП, подписки, аудитория-диаспора по миру, пользователей пока нет).

| Критерий | Paddle | Lemon Squeezy |
|---|---|---|
| Статус в 2026 | Зрелый, стабильный, доступен сразу | Куплен Stripe (2024), мигрирует в «Stripe Managed Payments» — для новых продавцов через waitlist, продукт в переходе |
| Базовая комиссия | 5% + $0.50 | 5% + $0.50 |
| Доп. сборы | минимум | +0.5% подписки, +1.5% международные, +1.5% PayPal, **+1% выплата на не-US счёт** |
| MoR / VAT ЕС | да (берёт на себя) | да |
| ИП без юрлица | да (тип Individual) | да |
| Выплаты | ежемесячно, минимум €100; SEPA EUR→EUR в еврозоне — без комиссии | быстрее (semi-monthly), но 1% на не-US счёт; есть жалобы на заморозку при ревью |
| Онбординг | верификация бизнеса + проверка домена (5–7 дней если ручная) | проще/быстрее |

**Почему Paddle:**
1. **Стабильность.** LS в середине слияния со Stripe — Managed Payments ещё не полностью открыт. Начинать новый бизнес на платформе «в переходе» — лишний риск. Paddle доступен и заточен под SaaS-подписки + VAT ЕС уже сейчас.
2. **Предсказуемость комиссий.** На твой кейс (подписки + международные клиенты + выплата на чешский/не-US счёт) у LS набегают надбавки. Paddle ровнее.
3. **Под ИП и под евро.** Тип Individual (без регистрации компании), выплаты SEPA в евро — без комиссии, если держать EUR-счёт.

**Практические нюансы:**
- Чехия — ЕС, но **не еврозона** (крона). Чтобы выплаты SEPA шли без комиссии — заведи EUR-счёт (например, Wise с европейским IBAN), получай в евро.
- Для верификации: документы ИП + живой сайт (`bydaria.kitchen` уже есть) + страницы Terms/Refund/Pricing.
- Минимум выплаты €100 — на старте баланс просто копится, это норма.
- Песочницу Paddle можно глянуть бесплатно хоть сейчас; реальный аккаунт — на этапе включения оплаты (гейт §8: ценность + аудитория), спешки нет.

> ⚠️ Это информационная рекомендация, не финансовый/налоговый совет. Финальную налоговую часть (OSS/VAT, как декларировать доход от Paddle) подтверди у чешского tax-advisor (€100–200 разово).

## 8. Порядок работ (фазировка)

> Привязка не к календарю, а к гейтам. Фазы A и B — **часы стройки** (вайб-кодинг ускоряет),
> Фаза C — **часы рынка** (не ускоряется: нужна аудитория). Старый ярлык «~месяц 10» снят.

**Фаза A — безопасный фундамент** (можно делать хоть сейчас, 0 риска, всё под флагом, для юзера
ничего не меняется). **Статус: частично.**
- ✅ `getEntitlements()` + фиче-флаг `MONETIZATION_ENABLED=false` (читает `profiles.plan`);
- 🔧 миграции: `profiles` +поля (`plan_expires_at`, `paddle_customer_id`), `subscriptions`,
  `credit_ledger` (+ RLS, `spend_credit()`) — **ещё не сделано**;
- 🔧 `PlanBanner` на реальные счётчики (пока хардкод Welcome-баланса).

**Фаза B — построить ценность** (это и есть «Premium-фичи»; по одной, сразу с метрингом
кредитов, но **без оплаты**). **Статус: идёт, опережает план.**
✅ своя книга рецептов (+ лимит) → ✅ пользовательский AI-КБЖУ (главный крючок) →
🔧 генерация рецепта → 🔧 импорт по URL → 🔧 экспорт PDF → 🔧 меню недели → 🔧 AI-картинки.

**Фаза C — включить деньги.** **Гейт (а не дата):** (1) ценность построена (Фаза B) **и**
(2) есть аудитория с доказанным retention — гейт Фазы 1 в `PRODUCT_STRATEGY` §7 (промо
проведено, в воронке есть кому платить). Тогда: Paddle (§5) + `MONETIZATION_ENABLED=true` +
paywall. Сам код Paddle можно написать заранее (часы стройки) — но переключатель денег
флипается только по гейту рынка.
