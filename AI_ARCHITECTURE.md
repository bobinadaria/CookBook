## 0. Принципы

Прежде чем нырять в детали, четыре аксиомы, которые определяют всю архитектуру:

**1. LLM не считает числа.** Claude и GPT галлюцинируют точные значения. Их роль — структурировать, классифицировать, переводить, генерировать креатив. Любое число (калории, граммы, минуты) должно приходить из **детерминированного источника** (USDA, Open Food Facts, формула BMR).

**2. AI-вызов = деньги.** Каждый запрос к Claude/OpenAI стоит реальные доллары. Архитектура должна минимизировать вызовы: кеширование, дешёвая модель там, где можно (Haiku), batch-запросы.

**3. Точность важнее скорости.** Пользователь, который один раз увидел «в твоей пасте 1200 ккал» вместо реальных 580 — теряет доверие навсегда. Лучше показать «не уверены, уточните вес» чем выдать неверное число.

**4. Дисклеймеры в UI, а не «врач рядом».** Мы не медицинский сервис. Все нутри-функции должны иметь видимый дисклеймер «не заменяет консультацию диетолога/врача». Это правовая защита в ЕС.

## 1. КБЖУ-калькулятор — главная AI-фича

Это сердце Premium-предложения. Здесь мы выигрываем у ChatGPT и оправдываем подписку. Архитектура подробно:

### 1.1 Общий поток

```
[Пользовательский ввод свободной формы]
        │
        ▼
[Step 1: Claude Haiku — парсинг ингредиентов в JSON-структуру]
        │
        ▼
[Step 2: Embedding-search в локальной БД продуктов USDA/OFF]
        │
        ▼
[Step 3: Маппинг ингредиента → продукт + confidence score]
        │
        ├─── confidence >= 0.85? ──→ [Step 4a: Прямой расчёт]
        │
        └─── confidence < 0.85? ───→ [Step 4b: Запрос уточнения у пользователя]
                                            │
                                            ▼
                                    [Step 5: Конверсия единиц → граммы]
                                            │
                                            ▼
                                    [Step 6: Поправка на способ готовки]
                                            │
                                            ▼
                                    [Step 7: Суммирование + деление на порции]
                                            │
                                            ▼
                                    [Step 8: Возврат КБЖУ-блока + confidence overall]
```

### 1.2 Step 1 — Парсинг ингредиентов через LLM

Вход: «2 ст. ложки оливкового масла, 200г куриной грудки, пол-стакана риса басмати»

Системный промпт (упрощённый):
```
Ты парсер кулинарных ингредиентов. Получаешь список ингредиентов на русском или английском.
Возвращаешь JSON-массив объектов вида:
{
  "raw_text": "оригинальная строка",
  "name_normalized": "нормализованное название (lowercase, без специй)",
  "name_en": "английский перевод для поиска в USDA",
  "amount": число,
  "unit": "g" | "ml" | "tbsp" | "tsp" | "cup" | "piece" | "slice",
  "cooking_method": "raw" | "boiled" | "fried" | "baked" | null,
  "is_optional": boolean,
  "is_garnish": boolean
}
```

Модель: **Claude Haiku 4.5**. Стоимость: ~$0.002 за рецепт из 10 ингредиентов. Скорость: <1 секунда.

Почему Haiku, не Sonnet: парсинг — простая задача, не требует «думать». Haiku справляется с точностью >97% на структурированных запросах. Использовать Sonnet здесь — переплата в 8 раз.

### 1.3 USDA FoodData Central — что это и зачем

**FoodData Central** — государственный американский API с базой ~400 000 продуктов. Бесплатный, без лимитов для разумного использования (требует API-ключ через https://api.nal.usda.gov/, выдаётся за 5 минут).

Структура данных одного продукта:
```json
{
  "fdcId": 173410,
  "description": "Chicken, broilers or fryers, breast, meat only, raw",
  "foodNutrients": [
    {"nutrientName": "Energy", "value": 114, "unitName": "kcal"},
    {"nutrientName": "Protein", "value": 21.23, "unitName": "g"},
    {"nutrientName": "Total lipid (fat)", "value": 2.62, "unitName": "g"},
    {"nutrientName": "Carbohydrate, by difference", "value": 0, "unitName": "g"}
  ],
  "servingSize": 100,
  "servingSizeUnit": "g"
}
```

Все значения — **на 100 г продукта** (это нормализованная база, удобная для расчётов).

**Слабые стороны USDA для нас:**
- Описания на английском (нужен перевод/маппинг)
- Уклон в американские продукты («Cheese, mozzarella, low moisture, part-skim») — для европейских пользователей часть продуктов отсутствует или искажена
- Не покрывает европейские/русские бренды («Простоквашино», «Активиа», «Останкинский»)

### 1.4 Open Food Facts — европейская альтернатива

**Open Food Facts** (https://world.openfoodfacts.org/data) — open-source база, ~3 миллиона продуктов с штрих-кодами, заметная доля европейских брендов.

Слабые стороны:
- Качество данных неравномерное (часть продуктов без полных данных по КБЖУ)
- Дубликаты
- Нужна аккуратная фильтрация по `nutriments_complete` флагу

**Сильные стороны:**
- Локализованные описания (есть русские, немецкие, французские)
- Реальные продукты из европейских магазинов
- Бесплатно, открыто, дамп можно загрузить целиком (~5 ГБ) и хостить локально

### 1.5 Гибридная стратегия маппинга

Наш собственный «product matcher» работает по слоям:

**Слой 1: Локальная БД базовых ингредиентов (мы делаем сами).**

Создаём таблицу `ingredients_base` в Supabase из ~5 000 наиболее частых ингредиентов, заранее замапленных и переведённых. Например:
```
{ name_ru: "куриная грудка", name_en: "chicken breast", usda_fdc_id: 173410,
  kcal_100g: 114, protein_100g: 21.23, fat_100g: 2.62, carbs_100g: 0 }
```

Это покрывает 80-90% запросов. Поиск — мгновенный, бесплатный.

**Слой 2: Embedding-search в USDA (если не нашли в локальной БД).**

Для не покрытых случаев — векторный поиск:
1. Берём `name_en` из парсера
2. Считаем embedding через `text-embedding-3-small` от OpenAI ($0.00002 за запрос)
3. Ищем ближайшие 5 продуктов в pre-computed USDA embeddings (один раз посчитанные, лежат в pgvector в Supabase)
4. Берём top-1, возвращаем confidence (косинусное расстояние)

**Слой 3: Open Food Facts для европейских специфичных продуктов.**

Если предыдущие слои дали confidence < 0.7 — пробуем Open Food Facts API. Это полезно для «адыгейский сыр», «творог 9%», «бородинский хлеб» — продуктов, которых нет в USDA.

**Слой 4: Запрос у пользователя.**

Если все слои провалились — UI показывает «Не нашли точного совпадения для "адыгейский сыр". Похоже на:
- (a) Сыр фета, 264 ккал
- (b) Сыр моцарелла, 280 ккал
- (c) Ввести данные вручную»

Пользователь выбирает или вводит — и его выбор **сохраняется в его персональный mapping**, чтобы в следующий раз сработал автоматически.

### 1.6 Конверсия единиц

Это нудная, но критическая часть. Пользователь пишет «пол-стакана риса», а USDA отвечает на 100 г. Нужны таблицы:

| Unit | Default conversion to grams |
|---|---|
| tbsp (масло) | 14 g |
| tbsp (сахар) | 12 g |
| tbsp (мука) | 8 g |
| tbsp (мёд) | 21 g |
| tsp | 5 g (вода-эквивалент) |
| cup (рис сухой) | 200 g |
| cup (мука) | 125 g |
| cup (молоко) | 240 g |
| slice (хлеб) | 25 g |
| piece (яйцо) | 50 g |
| piece (морковь) | 70 g |
| piece (картошка средняя) | 150 g |

**Стратегия:** держим таблицу `unit_conversions` в Supabase, индексирована по `(ingredient_id, unit)`. Для частых ингредиентов — точные значения. Для редких — fallback по категории (овощи / фрукты / крупы / молочка).

**Edge cases:** «по вкусу», «щепотка», «горсть» — игнорируем для КБЖУ-расчёта (это специи и зелень, КБЖУ-вклад <1%).

### 1.7 Поправка на способ приготовления

Сырая курица: 114 ккал/100г. Жареная в масле: ~250 ккал/100г (за счёт абсорбированного жира). Варёная: 165 ккал/100г (за счёт уплотнения после потери воды).

USDA содержит варианты для большинства продуктов:
- `Chicken breast, raw`
- `Chicken breast, boiled`
- `Chicken breast, roasted`

Парсер из 1.2 уже извлекает `cooking_method`. Мы маппимся на правильный вариант в USDA. Если варианта нет — применяем эвристику:

| Метод | Поправка |
|---|---|
| boiled | × 0.95 (потеря воды) |
| fried (с маслом) | + 5 g жира / 100 г блюда (стандартная абсорбция) |
| baked | × 0.93 |
| grilled | × 0.90 |
| raw | × 1.00 |

### 1.8 Confidence score и UX

Каждый расчёт сопровождается confidence score (0-1):

| Score | UI представление |
|---|---|
| >= 0.9 | «КБЖУ: 580 ккал» (без значка) |
| 0.7-0.9 | «КБЖУ: ~580 ккал ⓘ Точность ±10%, основано на близких продуктах» |
| 0.5-0.7 | «КБЖУ: ~580 ккал ⚠️ Хотите уточнить ингредиенты для большей точности?» |
| < 0.5 | «Не смогли точно посчитать. [Уточнить ингредиенты]» |

Confidence считается как среднее по всем ингредиентам, взвешенное по их массовой доле в блюде.

### 1.9 Пример полного расчёта

Ввод: «2 ст. ложки оливкового масла, 200г куриной грудки, 100г риса басмати, соль по вкусу, петрушка»

Парсинг:
```json
[
  { name_ru: "оливковое масло", amount: 2, unit: "tbsp", cooking_method: null },
  { name_ru: "куриная грудка", amount: 200, unit: "g", cooking_method: "fried" },
  { name_ru: "рис басмати", amount: 100, unit: "g", cooking_method: "boiled" },
  { name_ru: "соль", is_garnish: true },
  { name_ru: "петрушка", is_garnish: true }
]
```

Маппинг:
- Оливковое масло → локальная БД, 884 ккал/100г, conf 1.0
- Куриная грудка → локальная БД, 114 ккал/100г (raw), conf 1.0
- Рис басмати → embedding-search USDA, 130 ккал/100г (cooked), conf 0.88

Конверсия и поправки:
- Масло: 2 × 14 = 28 г → 884 × 0.28 = 248 ккал, 0 P, 28 г F, 0 C
- Курица: 200 г жареной = 200 × (114 + 5/100×200) /100 = 248 ккал (с учётом абсорбции 10 г масла = +88 ккал), 42.5 P, 13.2 F, 0 C
  - Уточнение: масло из ингредиентов уже учтено, поэтому абсорбция учитывается частично (50% от 28 г = 14 г масла абсорбируется курицей)
- Рис басмати: 100 г варёного × 1.30 = 130 ккал, 2.7 P, 0.3 F, 28 C
- Соль и петрушка: игнорируем (<1% вклад)

Итого по рецепту: ~626 ккал, 45 P, 41 F, 28 C
На 2 порции: **313 ккал, 22 P, 21 F, 14 C на порцию**
Confidence: 0.96 (средняя)

UI показывает:
```
🍽 313 ккал на порцию
  Б 22г · Ж 21г · У 14г
```

### 1.10 Где это лучше ChatGPT?

ChatGPT на этом же запросе вернёт что-то вроде «приблизительно 350-450 ккал», галлюцинируя. Наши 313 — основаны на реальных USDA-данных + детерминированной математике, точность ±5-8%. **Это measurable difference**, который можно показать пользователю на маркетинговой странице.

## 2. AI-генерация рецептов

### 2.1 Архитектура потока

```
[Запрос пользователя в свободной форме]
        │
        ▼
[Step 1: Claude Sonnet — генерация рецепта по структурированному промпту]
        │
        ▼
[Step 2: Парсинг и валидация JSON-ответа]
        │
        ▼
[Step 3: КБЖУ-калькулятор (Section 1) для ингредиентов]
        │
        ▼
[Step 4: Сохранение в БД с пометкой "ai-generated, не проверено пользователем"]
        │
        ▼
[Step 5: Отображение в UI с возможностью редактирования]
```

### 2.2 Системный промпт

Архитектура промпта (русская версия для русскоязычной аудитории):
```
Ты — кулинарный AI Cookbook. Генерируешь рецепты для домашней кухни на русском языке.

ОГРАНИЧЕНИЯ:
- Все ингредиенты должны быть доступны в обычном европейском супермаркете.
- Используй метрическую систему (граммы, мл), не cups/oz.
- Учитывай диету: {{diet_type}} (omnivore / vegetarian / vegan / keto / pescatarian / gluten_free)
- Калории на порцию: {{target_calories}} ±15%
- Аллергии: {{allergies}} (НЕ использовать)
- Не нравится: {{dislikes}} (избегать, можно заменить)
- Время готовки: до {{max_time}} минут

ФОРМАТ ОТВЕТА (строгий JSON):
{
  "title": "название рецепта",
  "description": "1-2 предложения о блюде",
  "servings": число,
  "prep_time_minutes": число,
  "cook_time_minutes": число,
  "ingredients": [
    { "name": "название", "amount": число, "unit": "g|ml|tbsp|tsp|piece" }
  ],
  "steps": [
    { "order": 1, "description": "шаг рецепта" }
  ],
  "tags": ["завтрак", "вегетарианское", ...]
}

ЗАПРЕЩЕНО:
- Указывать КБЖУ в ответе (это посчитается отдельно)
- Использовать сырые яйца в детских рецептах
- Рекомендовать сырое мясо/рыбу без предупреждения
- Делать медицинские заявления («лечит диабет», «улучшает работу сердца»)
```

Модель: **Claude Sonnet 4.6**. Стоимость: ~$0.015 за рецепт. Скорость: 3-5 секунд.

Почему Sonnet, не Opus: Sonnet справляется на 95%+, Opus добавляет ещё 2-3%, но стоит в 5 раз больше. Для пользовательской задачи разница незаметна.

### 2.3 Anti-hallucination layer

После генерации:
1. JSON-валидация (zod-схема). Если ответ не парсится — retry 1 раз с эксплицитным «Верни строго JSON по схеме».
2. Проверка ингредиентов на «существуют»: каждый ингредиент прогоняется через product matcher из Section 1.5. Если confidence < 0.5 — заменяем на ближайший существующий или флагуем для review.
3. Time-check: prep_time + cook_time не больше max_time.
4. Calories pre-check: грубая прикидка через сумму вкладов основных ингредиентов. Если получается +30% от target — retry с уточнением промпта.

### 2.4 Локализация рецептов

Двуязычная стратегия:
- Если интерфейс на русском → промпт на русском, ответ на русском, ингредиенты на русском.
- Если интерфейс на английском → промпт на английском, ответ на английском.
- БД хранит оба варианта в `recipes` через JSONB-поле `localized`.

Это требование к **Phase 2**, не MVP. На старте делаем только один язык (по выбору пользователя при регистрации).

### 2.5 Safety filter

Перед сохранением в БД:
- Проверка на «медицинские заявления» в описании (regex + LLM-классификатор)
- Проверка на «опасные» комбинации (сырое мясо для детей и т.д.) — простой rule-based
- Дисклеймер автоматически добавляется к каждому AI-рецепту в UI: «Сгенерировано AI. Проверьте на соответствие вашим диетическим ограничениям.»

### 3.1 Выбор модели

|---|---|---|---|---|
| DALL-E 3 standard | $0.040 | 6/10 | 5-8 сек | Слабый |
| DALL-E 3 HD | $0.080 | 7/10 | 10-15 сек | Слабый |
| **Flux Pro 1.1** (Replicate) | $0.055 | 9/10 | 5-10 сек | Хороший |
| Flux Schnell | $0.003 | 7/10 | 1-3 сек | Средний |

**Рекомендация:** **Flux Pro 1.1** через Replicate API.

- Понимает английские food-промпты лучше всех.

- 5-10 секунд — приемлемо для UX «нажал, налил кофе, готово».

**Альтернатива:** Flux Schnell для предпросмотра + Pro для финальной версии. Это сложнее, но дешевле для пользователя.

### 3.2 Стилевой промпт-каркас

Все картинки идут через один шаблон, который держит визуальную консистентность сайта:

```
{user_dish_description}, top-down view, warm natural light, wooden surface,
soft shadows, appetizing, cozy home kitchen aesthetic, film photography feel,
muted warm tones, depth of field, magazine quality food photography
```

С negative prompt:
```
plastic, fake, oversaturated, neon, dirty, messy, hands, faces, text, watermark,
cartoon, illustration, low quality, blurry
```

### 3.3 UX пресетов

Пользователь не пишет промпт сам. Видит 4 пресета:

| Пресет | Эстетика | Промпт-добавка |
|---|---|---|
| **Уют** | Тёплый, домашний | `warm tones, vintage, candlelight feel` |
| **Свежесть** | Светлый, прозрачный | `bright, airy, white background, fresh herbs` |
| **Драма** | Контрастный, тёмный | `moody dark background, low key lighting, dramatic shadows` |
| **Минимализм** | Чистый, скандинавский | `minimal composition, white plate, no garnish, scandinavian style` |

Пользователь:
1. Кликает «Сгенерировать фото»
2. Выбирает пресет
3. -1 кредит → 5-8 секунд ожидания (показываем skeleton с приятной анимацией)
4. Видит **4 варианта**
5. Выбирает один → сохраняется к рецепту → ещё -0 кредитов
6. Если не понравились — можно «Перегенерировать» (-1 ещё кредит)

### 3.4 Кредитная экономика

Уже описана в `PRODUCT_STRATEGY.md` Section 4. Для контекста:

- 1 «генерация» = 4 варианта = -1 кредит
- Пакеты: 50 / 120 / 300 кредитов
- Себестоимость на пакет S: 50 × $0.055 = $2.75 (за 4 варианта мы делаем 4 generations реально? или 1?)

**Решение архитектурное:** одна «генерация» делает **1 запрос к Flux с n=4 variants**. Flux Pro поддерживает batch с одним промптом — это даёт нам 4 варианта по цене 1.5× от единичного, не 4×. Точный API-параметр: `num_outputs: 4` в Replicate.

Себестоимость на пакет S: ≈ 50 × $0.08 (≈1.5× от $0.055) = **$4.00**.
Цена пакета S: €4.90 ≈ $5.30.
**Маржа после Paddle 5%: ~20%.** Тонко, но рабочая.

Решение: можно поднять цену S-пакета до €5.90 (50 картинок) или дать в S только 40 картинок за €4.90. Это финансовая оптимизация, можно подкрутить после запуска по фактическим данным.

## 4. Инфраструктура

### 4.1 Provider abstraction layer

Не привязываемся к одному провайдеру. Архитектура:

```
[App code]
   │
   ▼
[lib/ai/index.ts] ─── единый интерфейс
   │
   ├─→ [lib/ai/anthropic.ts] ─── Claude (текст)
   ├─→ [lib/ai/openai.ts] ─── GPT (опционально, fallback)
   ├─→ [lib/ai/replicate.ts] ─── Flux (картинки)
   └─→ [lib/ai/usda.ts] ─── USDA + OFF (данные)
```

Интерфейс:
```typescript
interface AIProvider {
  generateRecipe(params: RecipeParams): Promise<Recipe>;
  parseIngredients(text: string): Promise<ParsedIngredient[]>;
  generateImages(prompt: string, preset: ImagePreset): Promise<ImageUrl[]>;
}
```

Это позволяет в будущем легко переключиться, например, на свой fine-tuned модель или на дешёвый Cerebras/Groq inference.

### 4.2 Rate limiting

Два уровня:

**Уровень 1: на пользователя (бизнес-логика).**
- Free: 5 КБЖУ-расчётов и 3 рецепт-генераций в месяц.
- Premium: 200 любых AI-действий в сутки (soft-cap против абьюза).
- Картинки: всегда по кредитам, без daily-cap.

Реализация: счётчик в Supabase `user_ai_usage` с полями `month_kbju_count`, `month_recipe_count`, `today_total`. Sweep daily через cron.

**Уровень 2: на API (защита инфры).**
- Не больше 10 одновременных AI-запросов от одного юзера (предотвращает DDoS).
- Edge rate limit через Vercel: 60 запросов/мин/IP.

### 4.3 Caching

**Кеш для embedding-search:**
- USDA embeddings вычисляются один раз и хранятся в Supabase pgvector. Не перевычисляем.
- Embeddings ингредиентов из локальной БД — pre-computed batch (один раз при деплое БД).

**Кеш для AI-ответов:**
- Если пользователь сгенерировал «Паста карбонара на 400 ккал» — следующий пользователь с тем же запросом получает кешированный ответ. Hash от нормализованного промпта → response. TTL 30 дней.
- Это экономит 30-50% AI-вызовов в продакшене.

**Кеш для КБЖУ:**
- Хешируем `(ingredient_id, amount, unit, cooking_method)` → результат. TTL вечный (USDA-данные не меняются часто).

### 4.4 Cost monitoring

Дашборд (можно простой):
- Total Anthropic API spend (день / неделя / месяц)
- Total Replicate spend
- Per-user breakdown (топ-10 потребителей)
- Cost per active Premium user
- Cost per Free user (должен быть < €0.05/мес, иначе free-план под угрозой)

Реализация: ежедневный cron подтягивает спенды через API провайдеров, пишет в Supabase. Builder с Recharts на дашборде в `/admin/costs`.

**Алерты:**
- Anthropic spend > €15/день → email
- Анти-фрик: если один user сделал > 500 AI-действий за день → автоматический soft-suspend AI до подтверждения

## 5. Edge cases и safety

### 5.1 Аллергии и medical disclaimers

В пользовательских настройках хранится список аллергий. На каждом AI-генерированном рецепте перед сохранением проверяем:
- Содержит ли рецепт что-то из списка → если да, **не показываем** этот рецепт, генерируем заново.

Универсальный дисклеймер в footer pricing-страницы и onboarding:
> Cookbook не является медицинским сервисом. Все рекомендации по питанию носят информационный характер и не заменяют консультацию диетолога, нутрициолога или врача.

### 5.2 Дети и безопасность

Рецепты с сырыми яйцами (тирамису, безе с гоголь-моголь) — добавляется warning `Не рекомендуется детям до 5 лет и беременным`. Это автоматически добавляется промптом, если в ингредиентах есть `raw_egg: true`.

### 5.3 Чувствительные темы

Промпт явно запрещает:
- Экстремальные диеты (< 800 ккал/день)
- Голодание-протоколы
- Рекомендации для людей с пищевыми расстройствами

Если пользователь запрашивает что-то подобное («хочу 600 ккал в день, чтобы похудеть на 10 кг за месяц») — UI показывает мягкое сообщение «Такая цель может быть нездоровой. Рекомендуемый минимум — 1200 ккал. [Использовать 1200] [Хочу настроить вручную]».

### 5.4 Cultural и language nuance

«Творог» в России — это quark/farmer's cheese, не cottage cheese (хотя гугл-переводчик скажет «cottage cheese»). Подобных ловушек много. Решение:
- Локальная БД `ingredients_base` пишется вручную с правильными маппингами.
- LLM-парсер обучен на ru-промпте с явными правилами для трудных случаев.

Топ-10 «ловушек» для русско-английского food-маппинга:
| Русский термин | Правильный английский |
|---|---|
| творог | quark / farmer's cheese (НЕ cottage cheese) |
| сметана | sour cream (full-fat) |
| ряженка | ryazhenka / fermented baked milk |
| гречка | buckwheat groats |
| пельмени | dumplings filled with meat |
| компот | fruit compote drink |
| манка | semolina |
| перловка | pearl barley |
| куриная грудка | chicken breast (NOT chicken bosom!) |
| сосиски | hot dogs / frankfurters |

## 6. Что НЕ строим в MVP

| Фича | Почему не сейчас |
|---|---|
| AI-чат с нутрициологом | Юридически рискованно (medical advice), дорого в токенах, сложно в guardrails. Phase 3+. |
| Распознавание еды по фото | Компьютерное зрение для food — слабое (60-70% accuracy на домашних блюдах). Дорого. Phase 3+. |
| Voice input («Алиса, добавь рецепт») | Узкий сценарий, дорого, мало UX-выигрыша. Out of scope. |
| Автоматическое масштабирование рецепта на N порций | Это просто математика, делаем не AI, а кнопкой в UI. Не AI-фича. |
| Импорт рецепта по URL через AI-парсинг | Хорошая фича для Phase 2. Требует robust scraping + юридический анализ. |
| AI-меню на неделю | Phase 2. Сложный constraint-solver, нужен после данных по предпочтениям юзера. |
| Streaming AI-ответов с typewriter-эффектом | Cosmetic, не блокирует MVP. Phase 2 если будет время. |

## 8. Открытые вопросы для следующего раунда

1. **Локальная БД ингредиентов — собрать заранее или генерировать по мере спроса?** Заранее = 2-3 недели работы, но мгновенный продукт. По мере спроса = дешевле, но первые пользователи будут видеть «не нашли».
2. **Embedding-провайдер.** OpenAI ($0.00002/запрос) или открытая модель `nomic-embed-text` (бесплатно на своём инстансе)? Trade-off latency vs cost.
3. **Какой Flux endpoint?** Replicate vs fal.ai vs прямой Black Forest Labs API. Цена/латентность отличаются.
4. **Confidence threshold для UX.** Я предложил 0.85, но это субъективно — нужен user-testing.
5. **Multi-step recipes — нужна ли композиция (соус + основное блюдо)?** Это сложнее в БД и парсинге, но даёт «настоящие» сложные рецепты.
6. **i18n БД ингредиентов.** Если делаем bilingual — каждый ингредиент имеет ru+en название с момента создания, или мы переводим лениво при первом запросе?
