# Мобильная версия — аудит и план улучшений

> **Контекст.** Гипотеза: большинство пользователей придут с телефона, поэтому мобильный UX —
> приоритет №1 для понятности «о чём сайт» новичку и удобства использования.
> **Создано:** 2026-05-31. Аудит по коду (Tailwind mobile-first: классы без префикса = мобильный,
> `sm:`/`md:`/`lg:` = шире). Реальные скриншоты на телефоне нужно снять отдельно — см. §4.

---

## 0. Главное за 10 секунд

Мобильная версия **уже в хорошем состоянии** — это не стройка с нуля, а полировка.
Каркас сделан правильно: есть отдельный мобильный хедер с гамбургером, корректные тап-зоны,
поддержка «чёлки» iPhone, все сетки карточек и тарифов сворачиваются в одну колонку.

**Реально проблемная зона ровно одна — панель фильтров в каталоге** (горизонтальный скролл).
Остальное — мелкие косметические доработки. Не нужно переделывать всё; достаточно закрыть P0 + P1.

## 1. Что уже работает хорошо (не трогаем)

- **Хедер** (`Header.tsx`) — отдельный мобильный бар (`md:hidden`): логотип + язык + тема +
  гамбургер. Тап-зоны 44–48px (`h-11 w-11`, `min-h-[48px]`), анимация бургера, `pt-safe`/`pb-safe`
  под чёлку и home-indicator. Меню закрывается при смене страницы.
- **Viewport** (`app/layout.tsx`) — настроен правильно: `viewportFit: "cover"` (safe-area работает).
- **Сетки карточек** — рецепты, related, паки кредитов: `grid-cols-1 sm:grid-cols-2 lg:grid-cols-3`
  (на телефоне всегда одна колонка — читается отлично).
- **Тарифы** (`pricing`) — `grid-cols-1 lg:grid-cols-3`: на телефоне три тарифа стопкой, разделители
  переключаются с боковых на верхние. Сделано аккуратно.
- **Рецепт** — состав, шаги, блок КБЖУ, футер: все двухколоночные раскладки сворачиваются на телефоне,
  `lg:sticky` для состава игнорируется на мобиле. Корректно.
- **FilterDropdown** — панель ограничена шириной экрана (`maxWidth: calc(100vw - 16px)`), слушает
  `visualViewport` (правильно реагирует на клавиатуру iOS). Хорошая инженерия.

> Вывод: фундамент трогать не нужно. Дальше — точечные правки.

## 2. План правок (приоритезированный)

### P0 — сделать в первую очередь (бьёт по юзабилити)

**P0.1 — Панель фильтров в каталоге уезжает в горизонтальный скролл**
`src/app/(public)/recipes/page.tsx:177`
```
flex items-center gap-2 overflow-x-auto ... scrollbar-hide
```
На телефоне в один ряд встают: поиск (`w-44`) + «Напитки» + 4 выпадающих фильтра + «Сбросить» —
суммарно ~550px на экране 375px. Пользователь крутит вбок, не видит, какие фильтры активны.
Это главный барьер юзабилити на мобиле.

**Что сделать:** на телефоне заменить горизонтальный ряд на одну кнопку «Фильтры» (со счётчиком
активных), открывающую модалку/нижний drawer с вертикальным списком. Поиск оставить отдельной
строкой на всю ширину сверху. На `md+` — текущий ряд оставить как есть.

**P0.2 — Поле поиска раздувается на фокусе за пределы экрана**
`src/app/(public)/recipes/page.tsx:195`
```
w-44 ... focus:w-56
```
`w-56` = 224px; при фокусе на узком экране толкает соседние элементы / усиливает горизонтальный скролл.
**Что сделать:** на мобиле сделать поиск шириной во всю строку и убрать `focus:`-расширение
(`w-full md:w-44 md:focus:w-56`). Естественно решается вместе с P0.1.

### P1 — заметная полировка

**P1.1 — Блок КБЖУ: три макроса в три колонки тесно на узком экране**
`src/components/recipe/NutritionFacts.tsx:64`
```
mt-6 grid grid-cols-3 gap-5 sm:gap-6
```
Б/Ж/У в 3 колонки на 375px — каждая ~105px, цифры `text-[36px]`. Тесно, особенно с подписями.
**Что сделать:** проверить на реальном телефоне; при тесноте — `grid-cols-3` оставить только с `sm:`
(`grid-cols-1 sm:grid-cols-3` или 2-в-ряд). Низкий риск, лёгкая правка.

**P1.2 — Заголовок hero крупный на телефоне**
`src/app/(public)/page.tsx:34`
```
text-[clamp(3.25rem,8vw,120px)]  // минимум 52px даже на 375px
```
Минимум clamp — 52px, не масштабируется ниже. Длинные русские слова в заголовке могут переноситься
некрасиво или упираться в края. **Что сделать:** опустить нижнюю границу (например
`clamp(2.5rem,8vw,120px)`) и проверить перенос на длинных RU-строках.

**P1.3 — Hero: «о чём сайт» в первом экране**
`src/app/(public)/page.tsx:28` (`flex flex-col-reverse`)
На мобиле сначала идёт большое фото 4:3 (~280px), затем заголовок и подзаголовок. На коротком экране
(iPhone SE ~667px) ценностное сообщение и кнопка «Открыть рецепты» могут оказаться ниже сгиба.
**Что сделать:** снять на телефоне, как выглядит первый экран. Если фото съедает всё — чуть уменьшить
высоту фото на мобиле или поднять подзаголовок/CTA выше. Это про «новичок за 5 сек понимает, что это».

### P2 — мелочи / на потом

- **P2.1** Хлебные крошки на странице рецепта (`recipes/[slug]/page.tsx`): «← К рецептам» + категория +
  звезда избранного в один ряд — на узких экранах тесновато. Можно сократить текст до «←».
- **P2.2** Метрики рецепта (`:289`, `grid-cols-2`) — 2×2 на мобиле, в целом ок; глянуть вживую.
- **P2.3** Pull-quote (`text-[26px]` на мобиле) — частые переносы на 375px; вопрос вкуса, не баг.
- **P2.4** (Опционально, не обязательно) нижняя таб-навигация на мобиле (Главная / Рецепты / Книга).
  Гамбургер и так работает — это апгрейд, а не фикс. Делать только если захочется.

## 4. Чек-лист ручного тестирования на телефоне (то самое «тестирование mobile ui»)

Открыть `bydaria.kitchen` на реальном телефоне (или DevTools → iPhone SE 375px / iPhone 14 390px),
пройти и отметить:

**Понятность для новичка**
- [ ] На первом экране главной за ~5 сек понятно, что это за сайт (личная кулинарная книга + AI-КБЖУ)?
- [ ] Видно, куда нажать дальше (кнопка «Открыть рецепты» / меню)?

**Навигация**
- [ ] Гамбургер открывается/закрывается, пункты нажимаются пальцем без промаха?
- [ ] Переключение языка RU/EN и темы работает на мобиле?

**Каталог рецептов**
- [ ] Нет горизонтального скролла страницы?
- [ ] Фильтры понятно открываются, видно какие активны, легко сбросить?
- [ ] Поиск удобно вводить, экран не «прыгает» при фокусе?

**Страница рецепта**
- [ ] Заголовок и фото помещаются, без обрезки?
- [ ] Состав → шаги → КБЖУ читаются, цифры Б/Ж/У не слипаются?
- [ ] Кнопка «избранное» нажимается?

**Подписка / тарифы**
- [ ] Три тарифа идут стопкой, всё читается, кнопки на всю ширину?

**Общее**
- [ ] Нигде нет горизонтального скролла и текста, упирающегося в края?
- [ ] Ничего не «вылезает» при повороте экрана?

> Снять 4–5 скриншотов проблемных мест и приложить к задаче — этого хватит, чтобы решить, что чинить.

## 5. Важная оговорка

Этот аудит сделан **по коду** (для Tailwind это точно отражает мобильную раскладку), а не по живым
скриншотам: окно браузера в этой сессии было залочено на десктопной ширине и не сжималось до телефона.
Поэтому пункты с пометкой «проверить на телефоне» (P1.1, P1.3, P2.2) нужно подтвердить глазами на
реальном устройстве — это и есть «тестирование mobile ui» из чек-листа §4.
